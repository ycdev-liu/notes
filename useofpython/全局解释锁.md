**全局解释器锁（GIL, Global Interpreter Lock）** 是 Python（准确说是 **CPython 解释器**）中的一个机制：

> **在同一时刻只允许一个线程执行 Python 字节码。**

即使你创建了多个线程，它们也不能真正同时执行 Python 代码。

---

# 一、为什么会有 GIL

GIL 的主要原因是 **简化内存管理**。

Python 使用 **引用计数（reference counting）** 来做垃圾回收：

```text
对象引用 +1
对象引用 -1
引用为 0 → 回收
```

如果多个线程同时修改引用计数：

```text
线程A: count +1
线程B: count -1
```

可能会导致 **数据竞争（race condition）**。

为了避免这种问题，CPython 加了一个 **全局锁**：

```text
执行 Python 代码之前
必须先获得 GIL
```

---

# 二、GIL 的工作方式

大致流程：

```text
线程1 获取 GIL
↓
执行一段 Python 字节码
↓
释放 GIL
↓
线程2 获取 GIL
↓
执行代码
```

Python 会 **定期切换线程**（默认约每 5ms）。

可以理解为：

```text
线程轮流执行
而不是同时执行
```

---

# 三、GIL 对性能的影响

## 1 CPU密集任务（受影响）

例如：

* 数值计算
* 图像处理
* AI训练
* 加密运算

示例：

```python
import threading

def calc():
    for i in range(10**8):
        pass

t1 = threading.Thread(target=calc)
t2 = threading.Thread(target=calc)

t1.start()
t2.start()
```

结果：

```text
两个线程 ≈ 一个线程速度
```

因为 **GIL 限制只能一个线程执行**。

---

## 2 IO密集任务（影响不大）

例如：

* 网络请求
* 数据库
* 文件读取
* 爬虫

当线程执行 IO 时：

```text
线程等待 IO
↓
释放 GIL
↓
其他线程执行
```

所以：

```text
IO任务 → 多线程非常有效
```

---

# 四、解决 GIL 的方法

## 1 使用多进程（最常见）

```python
from multiprocessing import Process
```

原因：

```text
每个进程都有独立 GIL
```

CPU 可以真正并行。

---

## 2 使用 C 扩展

例如：

* `numpy`
* `pandas`
* `pytorch`

很多 C 扩展会 **释放 GIL**。

---

## 3 使用协程（asyncio）

适合：

```text
高并发 IO
```

例如 Web 服务器。

---

# 五、GIL 的真实影响总结

| 场景    | 推荐              |
| ----- | --------------- |
| CPU密集 | multiprocessing |
| IO密集  | threading       |
| 高并发网络 | asyncio         |

# Python 并发模型结构图

```
                    操作系统 CPU
                 ┌───────────────┐
                 │   多核 CPU     │
                 └──────┬────────┘
                        │
                多进程（真正并行）
                        │
            ┌───────────┴───────────┐
            │                       │
        Process 1               Process 2
   (独立内存 + 独立GIL)     (独立内存 + 独立GIL)
            │                       │
        多线程                   多线程
            │
    ┌───────┴────────┐
    │                │
 Thread 1        Thread 2
        \          /
         \        /
          ─── GIL ───
    同一时间只允许一个线程执行
```

关键理解：

```
CPU
 └─ 进程 (multiprocessing)  ← 可以真正并行
      └─ 线程 (threading)   ← 受 GIL 限制
           └─ 协程 (asyncio) ← 单线程并发
```

# 一、三种并发方式对比

| 并发方式 | 模块                | 是否并行     | 适合任务  |
| ---- | ----------------- | -------- | ----- |
| 进程   | `multiprocessing` | ✅ 真并行    | CPU密集 |
| 线程   | `threading`       | ❌ 受GIL限制 | IO密集  |
| 协程   | `asyncio`         | ❌ 单线程    | 高并发IO |

---

# 二、直观理解三者

### 1 进程（multiprocessing）

```
CPU core1 → process1
CPU core2 → process2
CPU core3 → process3
```

特点：

* 每个进程 **独立内存**
* 每个进程 **独立 GIL**
* 可以利用 **多核 CPU**

适合：

```
数据计算
机器学习
图像处理
```

---

### 2 线程（threading）

```
process
 ├── thread1
 ├── thread2
 └── thread3
        ↓
       GIL
```

特点：

* 共享内存
* 创建成本低
* 受 GIL 限制

适合：

```
网络请求
数据库
爬虫
文件读写
```

---

### 3 协程（asyncio）

```
thread
 └── event loop
      ├── task1
      ├── task2
      └── task3
```

特点：

* 单线程
* 无线程切换开销
* 超高并发

适合：

```
Web服务器
API
爬虫
高并发服务
```

很多现代框架使用协程，例如：

* FastAPI
* Sanic

---

# 三、最常见实际组合

真实项目通常是：

```
多进程
   ↓
每个进程运行一个 async web server
   ↓
asyncio 处理并发请求
```

例如 Web 服务：

```
gunicorn
   ├─ process1
   ├─ process2
   ├─ process3
        ↓
      asyncio
        ↓
   thousands requests
```

> **为什么 FastAPI 比 Flask 快很多？**

这个问题本质就是 **线程 vs 协程 vs GIL**。



很好 👍 这个问题其实是 Python Web 面试里 **非常经典的一题**：

> **为什么 FastAPI 通常比 Flask 更快？**

核心原因其实是 **并发模型不同：线程 vs 协程**。

---

# 一、Flask 的并发模型（WSGI）

Flask基于 **WSGI**（Web Server Gateway Interface）。

典型结构：

```text
client request
      ↓
web server (gunicorn / uwsgi)
      ↓
线程 / 进程
      ↓
Flask
```

处理方式：

```text
一个请求
  ↓
一个线程
```

示意：

```
thread1 → request1
thread2 → request2
thread3 → request3
```

问题：

* 线程切换成本高
* 线程数量有限
* 受 **GIL** 影响

当并发很多时：

```
10000 requests
↓
需要大量线程
↓
系统开销巨大
```

---

# 二、FastAPI 的并发模型（ASGI）

FastAPI 使用 **ASGI**（Asynchronous Server Gateway Interface）。

典型运行在：

* Uvicorn
* Hypercorn

结构：

```text
client request
      ↓
ASGI server
      ↓
event loop
      ↓
coroutine
      ↓
FastAPI
```

处理方式：

```text
一个线程
可以处理成千上万请求
```

示意：

```
event loop
 ├─ request1 coroutine
 ├─ request2 coroutine
 ├─ request3 coroutine
 └─ request10000 coroutine
```

因为 **协程是非阻塞的**。

---

# 三、核心区别

| 特性   | Flask | FastAPI |
| ---- | ----- | ------- |
| 接口标准 | WSGI  | ASGI    |
| 并发模型 | 线程    | 协程      |
| 并发能力 | 几百    | 几万      |
| IO效率 | 一般    | 非常高     |

---

# 四、举一个真实场景

假设 API 需要访问数据库：

### Flask

```python
def api():
    data = db.query()
    return data
```

执行流程：

```
线程
↓
等待数据库
↓
线程被阻塞
```

---

### FastAPI

```python
async def api():
    data = await db.query()
    return data
```

执行流程：

```
协程
↓
等待数据库
↓
切换到其他请求
```

线程不会被浪费。

---

# 五、性能差距示意

假设：

```
10000 并发请求
```

Flask：

```
需要大量线程
线程切换成本高
内存占用大
```

FastAPI：

```
单线程
event loop
协程调度
```

通常测试结果：

```
Flask   → 2k~5k rps
FastAPI → 20k~50k rps
```

（取决于业务）

---

# 六、但很多人误解的一点

**FastAPI 不一定总是更快。**

如果是 **CPU密集任务**：

```
图像处理
机器学习
数据计算
```

差别几乎没有。

因为：

```
CPU计算
↓
仍然受 GIL 限制
```

       uvicorn
